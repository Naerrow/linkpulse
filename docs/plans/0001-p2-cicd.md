## 1. 배경·목표

P1 실가동 검증이 전부 통과했다(ECS 2/2, 이미지 `v1`, DB_* 분리주입+RDS 실연결, healthz/readyz 200, 영속성 10/10). 두 가지를 P2에서 해소한다:

1. **자동 배포 부재** — 코드 변경마다 사람이 build/push/`terraform apply -var ...`를 손으로 했다.
2. **tfvars drift 위험** — `terraform.tfvars`(gitignore·자동 로드)에 `service_desired_count=0`·`image_tag="bootstrap"`이 남아 있어, 무인자 `terraform apply`가 prod를 0으로 죽일 수 있었다.

## 2. 계획 (확정 결정)

- **인증**: GitHub OIDC 페더레이션(장기 키 없음). **트리거**: main 푸시 자동 + `workflow_dispatch`(수동/롤백). **PR은 검증만.**
- **분리 원칙(가드레일 #1 정합)**: 인프라=Terraform(**사람 `plan`→`apply`**), 앱 이미지 배포=CI(**aws CLI**: register-task-definition + update-service). **CI는 `terraform apply`를 실행하지 않는다.**
- 경계 장치: `aws_ecs_service.app`에 `lifecycle { ignore_changes = [task_definition] }`.

## 3. 이행 기록 (Task별)

| # | 작업 | 파일 | 핵심 내용 |
|---|---|---|---|
| 1 | drift 잠금 | `terraform.tfvars`(로컬·미커밋), `variables.tf`, `terraform.tfvars.example` | `service_desired_count` default 0→2, 로컬 tfvars `=2`/`image_tag="v1"`, example은 운영 안전값+bootstrap 예외 주석 |
| 2 | OIDC IAM | `github_oidc.tf`(신규), `variables.tf` | OIDC provider(중복방지 토글 `existing_github_oidc_provider_arn`), 배포 role `linkpulse-prod-gha-deploy`(main exact trust), 최소권한 정책, `data.aws_caller_identity` |
| 3 | 경계·출력 | `ecs.tf`, `ecr.tf`, `outputs.tf` | service `ignore_changes=[task_definition]`+함정 주석, ECR 보관 10→30, output 3개(`github_actions_role_arn`/`ecs_task_definition_family`/`ecr_repository_name`) |
| 4 | 문서화 | `docs/adr/0001`(신규), `infra/README.md` | Terraform↔CI 경계 ADR, README에 경계 단락 + bootstrap 순서 재작성 |
| 5 | ci.yml | `.github/workflows/ci.yml`(신규) | `pull_request`+`workflow_call`, setup-go(`go-version-file`+`app/go.sum` 캐시), `make vet/test`+gofmt+Docker smoke(PR만) |
| 6 | deploy.yml | `.github/workflows/deploy.yml`(신규) | main push+dispatch, OIDC, `setup`/`checks`/`deploy` 3-job, ECR preflight 멱등, arm64 빌드·push(sha), render+deploy(wait-for-service-stability), 롤백 input |
| 7 | 로컬 검증 | — | `terraform fmt/validate`, `go vet/test`, 워크플로 YAML 문법 |

### IAM 최소권한 형태(배포 role)
- ECR: `GetAuthorizationToken`(`*`) + push/pull + `DescribeImages`(preflight) → repo ARN
- ECS: `RegisterTaskDefinition` → `task-definition/linkpulse-prod-app:*` family ARN · `DescribeTaskDefinition` → `*` · `UpdateService`/`DescribeServices` → service ARN
- `iam:PassRole` → execution/task 2개 role(+ `iam:PassedToService=ecs-tasks.amazonaws.com`)

### 배포 파이프라인 흐름
`setup`(태그·빌드여부 결정) → `checks`(롤백이면 skip, 아니면 ci.yml 재사용) → `deploy`(preflight 멱등 → arm64 빌드·push → family 최신 revision base로 render → update-service 롤링 → 안정화 대기). 롤백 = `workflow_dispatch`에 과거 sha 입력(빌드 skip, ECR 보관 30개 범위 내).

## 4. 교차검토 반영 (5라운드)

여러 AI 교차검토를 거쳤다. 핵심 수용/거부:

**수용**
- tfvars는 gitignore라 **편집만·`git add` 금지**(default 변경만으론 drift 미해결 — 자동 로드되어 default를 덮어쓰기 때문).
- OIDC provider 계정당 중복(EntityAlreadyExists) → 토글 변수로 회피.
- `ignore_changes` 함정(Terraform taskdef 변경은 CI 배포 1회 후 라이브 반영) → ADR/주석 명시.
- `aws ecr describe-images`는 URL이 아니라 `--repository-name` → `ECR_REPOSITORY_NAME` 분리.
- monorepo `cache-dependency-path: app/go.sum`, `go-version-file: app/go.mod`.
- `workflow_dispatch` 빈 input=신규/non-empty=롤백, tag regex 검증+shell quote, `deploy` job `if: github.ref=='refs/heads/main'` guard.
- **bootstrap 절차 재작성**: default가 2가 되고 `ignore_changes`가 생겨, 최초 생성은 `-var service_desired_count=0` → CI가 첫 이미지 배포 → `apply`로 desired=2.
- `image_tag`는 **baseline/초기 태그**(ignore_changes라 `-var`로 라이브 전환 안 됨); 비상 배포·롤백은 CI 또는 `aws ecs update-service`.
- (추가 라운드) IAM `UpdateService`/`DescribeServices` resource를 `aws_ecs_service.app.id`→**`.arn`**(id 모호성 회피, `terraform validate`로 attribute 입증). `infra/README.md`의 `terraform output`에 **`-raw`**(따옴표 없는 값으로 GitHub Variables 복사). 루트 `README.md` 배포 순서를 P2 기준(이미지=CI, `apply`=인프라·스케일)으로. `deploy` job `if`를 `build_image=='false' || checks success`로 의도 명확화. `jq`에 `deregisteredAt` 방어, `go-version-file: app/go.mod`.

**거부(근거 들어 정정)**
- "`ecs:RegisterTaskDefinition`은 `*`만 가능" 주장은 **틀림**. AWS Service Authorization Reference 원문을 직접 확인한 결과 `RegisterTaskDefinition`은 **`task-definition` 리소스 타입을 지원**(family ARN 제한 가능). → family ARN 유지. 단 ADR에 "첫 배포에서 AccessDenied 시 `*`로 완화" 폴백 명시.
- 교훈: 단정적 IAM/AWS 주장은 추측 말고 1차 출처로 확인한다.

## 6. 현재 상태 · 남은 사용자 단계

스테이징됨(`terraform.tfvars` 제외): `.github/workflows/ci.yml`·`deploy.yml`, `docs/adr/0001`, `docs/plans/0001`(이 문서), `README.md`, `infra/README.md`, `infra/prod/{github_oidc,ecs,ecr,outputs,variables}.tf`, `terraform.tfvars.example`.

> **주의**: 현재 스택은 이미 2태스크/v1로 운영 중이므로, 이번 apply는 README의 신규 bootstrap 절차가 아니라 **추가·전환**이다(desired no-op, task_definition ignore 시작, OIDC/lifecycle/outputs 추가, 파괴 없음).

1. apply 전: `aws iam list-open-id-connect-providers`로 OIDC 중복 확인(있으면 `existing_github_oidc_provider_arn` 지정) + 라이브 이미지 태그가 tfvars `image_tag`와 일치 확인.
2. `terraform plan`(파괴 없는지 확인) → `terraform apply`.
3. `terraform output` 6개 → GitHub repo Variables 등록: `AWS_DEPLOY_ROLE_ARN`/`ECR_REPOSITORY_URL`/`ECR_REPOSITORY_NAME`/`ECS_CLUSTER_NAME`/`ECS_SERVICE_NAME`/`ECS_TASK_DEFINITION_FAMILY`.
4. 커밋·푸시(새 브랜치 `feat/p2-cicd` 권장).
5. main 푸시/PR/`workflow_dispatch`로 파이프라인 가동 → 배포 후 P1 검증 블록 재실행 + `workflow_dispatch` 롤백 1회 검증.
